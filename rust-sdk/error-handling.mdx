title: "Error Handling"
description: "Handle Wacht API errors and SDK configuration issues cleanly."
---

The Wacht SDK exposes a unified `wacht::Error` enum that encapsulates network initialization failures, configuration issues, and actual HTTP API errors returned by your Wacht backend.

## The Error Enum

Almost all async SDK methods return `Result<T, wacht::Error>`.

```rust
#[derive(Debug, thiserror::Error)]
pub enum Error {
    /// API errors containing a specific status code and error model
    #[error("API error ({status}): {message}")]
    Api {
        status: axum::http::StatusCode,
        message: String,
        details: Option<serde_json::Value>,
    },

    /// Reqwest networking errors (DNS failures, connection drops)
    #[error("Request error: {0}")]
    Request(#[from] reqwest::Error),

    /// Missing environment variables or malformed configuration
    #[error("Configuration error: {0}")]
    Config(String),
}
```

## Handling API Responses

The most common error scenario you'll need to handle is `Error::Api`, which indicates the request successfully reached the Wacht backend but was rejected (e.g., 400 Bad Request, 404 Not Found, 429 Too Many Requests).

You can pattern match against the enum variant to extract the HTTP status code and apply specific fallback logic.

```rust
use axum::http::StatusCode;
use wacht::{WachtClient, Error};

async fn disable_user(client: &WachtClient, user_id: &str) -> Result<(), Error> {
    match client.users().user_id_patch(user_id).send().await {
        Ok(_) => {
            println!("Successfully disabled user.");
            Ok(())
        },
        Err(Error::Api { status, message, .. }) if status == StatusCode::NOT_FOUND => {
            // The user didn't exist in Wacht, so we can ignore the error
            println!("User already deleted or not found: {}", message);
            Ok(())
        },
        Err(Error::Api { status, message, details }) if status == StatusCode::TOO_MANY_REQUESTS => {
            // Rate limits triggered, wait and retry
            let retry_after = details.unwrap().get("retry_after").unwrap().as_u64();
            println!("Rate limited. Wait {} seconds.", retry_after);
            Err(Error::Api { status, message, details: None })
        }
        Err(e) => {
             // Some other network or authentication error
             Err(e)
        }
    }
}
```

## Integrating with Axum

If you are writing an Axum backend and want to transparently propagate Wacht API errors down to your own clients, you can implement Axum's `IntoResponse` trait on a custom application error wrapper.

```rust
use axum::{
    response::{IntoResponse, Response},
    http::StatusCode,
    Json,
};
use serde_json::json;

pub enum AppError {
    Wacht(wacht::Error),
    // ... your other errors
}

// Convert inner Wacht errors into HTTP responses
impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        match self {
            AppError::Wacht(wacht::Error::Api { status, message, .. }) => {
                // Return exactly the status code Wacht returned
                (status, Json(json!({ "error": message }))).into_response()
            }
            AppError::Wacht(_) => {
                // Hide internal network/config errors as 500s
                (StatusCode::INTERNAL_SERVER_ERROR, "Internal server error".to_string()).into_response()
            }
        }
    }
}

// Implement From so we can use the `?` operator directly in handlers
impl From<wacht::Error> for AppError {
    fn from(err: wacht::Error) -> Self {
        AppError::Wacht(err)
    }
}
```
