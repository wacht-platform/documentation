Just as you subscribe to Wacht's webhooks (e.g., to know when a new user signs up), your own B2B customers likely want to subscribe to *your* webhooks (e.g., when a project completes processing).

Wacht provides **Webhook Apps** to act as a robust, fully-managed outward webhook dispatcher for your platform. Wacht handles endpoint management, payload signing, retries with exponential backoff, and delivery analytics.

## Core Concepts

When you create a Webhook App within a Wacht deployment, it acts as a container for your customers' endpoints. 

1. **You define a Catalog**: Using the Wacht backend, you declare a static list of events your platform can emit (e.g., `invoice.paid`, `report.generated`).
2. **Customers create Endpoints**: Customers provide destination URLs and subscribe to specific events from your catalog.
3. **You dispatch Events**: Your backend sends a single API call to Wacht containing the event data.
4. **Wacht handles Delivery**: Wacht fans out the webhook to all subscribed endpoints, applying HMAC signatures and managing failure retries automatically.

## Implementing a Webhook Portal

Allowing your users to manage their webhook endpoints requires a secure handoff mechanism so your frontend can communicate with Wacht on the user's behalf. 

Similar to API Keys, Wacht handles this using a [Session Ticket Exchange](/backend-api/session-tickets):

1. The User wants to manage their Webhook endpoints in your frontend UI.
2. Your frontend requests an access ticket from your backend.
3. Your backend creates a `webhook_app_access` Ticket using its Secret API Key and passes it back.
4. Your frontend "exchanges" the ticket with the Wacht Frontend API to receive an elevated session.

With this elevated session, your frontend interface can query Wacht directly to create, read, update, and delete the user's webhook endpoints without proxying the requests through your own backend.

## Frontend API Highlights

The Wacht Frontend API (`https://api.wacht.dev/v1/webhook`) provides full endpoint management for the user's browser.

### Managing Endpoints

Users can configure URLs, add custom headers (like their own authorization strings), and select event subscriptions.

```http
POST /webhook/endpoints
```
```json
{
  "url": "https://customer-api.com/incoming-hooks",
  "description": "Payment Sync",
  "subscriptions": ["invoice.paid", "invoice.failed"]
}
```

Endpoint deliveries automatically fail over. If an endpoint fails to return a `2xx` response code for 10 consecutive attempts (across the exponential backoff curve), Wacht will gracefully disable the endpoint. 

### Delivery Inspection & Replay

Wacht logs every delivery attempt, including the exact headers and bodies sent and received. The frontend API allows your users to inspect these logs to debug their own integrations.

Users can also initiate bulk replays if their target system experienced an outage:

```http
POST /webhook/deliveries/replay
```
```json
{
  "start_date": "2024-01-01T00:00:00Z",
  "end_date": "2024-01-02T00:00:00Z",
  "status": "failed"
}
```

## Security & Payload Verification

When Wacht delivers a webhook to your target endpoint, it signs the payload using HMAC-SHA256 based on the unique `signing_secret` assigned to the Webhook App. 

Wacht implements the [Standard Webhooks](https://www.standardwebhooks.com/) specification (`webhook-id`, `webhook-timestamp`, `webhook-signature` headers). 

Users can trigger a secret rotation at any time via the frontend API if they believe their secret has been compromised.

```http
POST /webhook/rotate-secret
```

This immediately invalidates the old secret and begins signing all new deliveries with the generated secret.
