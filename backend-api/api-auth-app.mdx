title: "API Auth Apps"

While you use Wacht's backend Secret API keys to authenticate your own services, you often need to issue API keys to *your* users so they can interact programmatically with your application.

Wacht provides **API Auth Apps** to handle this completely out-of-the-box. An API Auth App represents a designated surface area of your API that your customers can generate keys for.

## Core Concepts

When you create an API Auth App within a Wacht deployment, Wacht automatically handles:
- **Key Generation & Storage**: Wacht securely generates, hashes, and stores the keys. You only ever see the prefix and suffix after creation.
- **Revocation & Rotation**: Wacht provides endpoints for rotating and revoking compromised keys.
- **Rate Limiting**: You can associate custom rate-limits with an API App, and Wacht will enforce them automatically at the Gateway.
- **Audit Logging**: Every time an API key is used, Wacht can log its usage, providing usage metrics back to the end-user.

## Implementing an API Key Portal

To allow your users to generate API Keys, they must authenticate with your application (thereby establishing a Wacht session token). 

Because managing API Keys requires elevated privileges, your frontend cannot mint API keys directly. Instead, you use a [Session Ticket Exchange](/backend-api/session-tickets) pattern:

1. The User requests access to the API Key Portal in your frontend UI.
2. Your frontend sends the request to your backend.
3. Your backend validates the user's permissions, and if authorized, requests a `api_auth_access` Ticket from Wacht using its Secret API Key.
4. Your backend passes this short-lived Ticket back to your frontend.
5. Your frontend "exchanges" the ticket with the Wacht Frontend API to receive an elevated session, allowing it to interface directly with the API Auth App endpoints shown below.

This pattern ensures your backend remains the ultimate source of permission, while offloading the complexity of key management UI entirely to Wacht.

## Frontend API Endpoints

Once the frontend has exchanged a ticket for an elevated session, it can directly query the Wacht Frontend API (`https://api.wacht.dev/v1/api-auth`) on behalf of the user to manage their keys. 

*Note: These endpoints are intended to be called by the browser, not your backend.*

### Listing API Keys

```http
GET /api-auth/keys?status=active
```

Returns a list of the user's generated API Keys. Due to security, the `key` field is never returned in list responses—only metadata such as `name`, `key_prefix`, and `expires_at`.

### Creating an API Key

```http
POST /api-auth/keys
```

```json
{
  "name": "Production Server Key",
  "expires_at": "2024-12-31T23:59:59Z" // Optional
}
```

This is the **only** time the full `key` string will be returned. Make sure to instruct your user to copy it immediately.

### Rotating an API Key

```http
POST /api-auth/keys/{id}/rotate
```

Generates a new underlying secret for the key while preserving its metadata, permissions, and ID. This invalidates the old secret immediately.

### Revoking an API Key

```http
POST /api-auth/keys/{id}/revoke
```

```json
{
  "reason": "Compromised by accidental github commit"
}
```

Instantly invalidates the API Key. All subsequent requests using this key will be rejected at the Wacht Gateway.

## Analytics & Audit Logs

Wacht tracks every request made using an API key generated through an API Auth App. Your frontend can query these logs to provide a dashboard to your users.

```http
GET /api-auth/audit/analytics
```

This endpoint returns total request volumes, success rates, top paths accessed, and rate-limit hits, allowing you to easily build rich usage visualizations for your B2B customers.
